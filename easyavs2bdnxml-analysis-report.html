1. 项目概览与生态定位

1.1 什么是 easyavs2bdnxml

easyavs2bdnxml 是一个 Windows 平台的字幕格式转换工具，用于将文本字幕（ASS/SSA/SRT）转换为蓝光光盘兼容的图形字幕格式 SUP (PGS — Presentation Graphic Stream)。它是 avs2bdnxml 命令行工具的 GUI 封装，集成了完整的渲染和编码管线。

1.2 生态系统中的位置

输入层 / Input
  • ASS/SSA — Advanced SubStation Alpha
  • SRT — SubRip 文本字幕
  • AviSynth 脚本 — RGBA 帧序列
处理层 / Processing
  • easyavs2bdnxml — GUI 编排器
  • AviSynth 2.60 — 脚本引擎
  • VSFilter / VSFilterMod — 字幕渲染
  • avs2bdnxml — RGBA→BDN XML+PNG
  • BDSup2Sub — BDN XML→SUP
输出层 / Output
  • SUP — PGS (Presentation Graphic Stream)
  • BDN XML + PNG — 中间格式
  • PES+MUI — 专业 Authoring 格式

2.2 各阶段详细流程

阶段 1：AviSynth 脚本生成

easyavs2bdnxml 自动生成 .avs 脚本，核心内容：

// 自动创建的 AviSynth 脚本
LoadPlugin("VSFilter.dll")      // 或 VSFilterMod.dll
LoadPlugin("DirectShowSource.dll")
video = DirectShowSource("input.mkv")
sub = TextSubMod("input.ass")      // 加载 ASS 字幕
Overlay(video, sub)               // 合成
ConvertToRGB32(video)             // 转为 RGBA

阶段 2：VSFilter 渲染

VSFilter 是 DirectShow 字幕过滤器，负责将 ASS 文本指令渲染为 RGBA 位图：

  • 解析 ASS 文件的 [Script Info]、[V4+ Styles]、[Events] 段
  • 按时间戳逐帧渲染字幕，处理字体、颜色、位置、动画
  • 输出 32-bit RGBA 位图帧序列
  • VSFilterMod 扩展支持更多特效（模糊、描边等）

阶段 3：avs2bdnxml 转换

核心逻辑——将 RGBA 帧序列转为 BDN XML + PNG 文件：

// avs2bdnxml 核心算法伪代码
for each_frame(frame_idx):
    rgba_frame = avs_env.get_frame(frame_idx)  // 获取 RGBA 帧
    
    // 1. 检测帧是否有字幕内容（非全透明）
    if is_empty(rgba_frame):
        continue
    
    // 2. 自动裁剪边界（去除透明区域）
    cropped = autocrop(rgba_frame)
    
    // 3. 量化为 8-bit 调色板 PNG（可选）
    if palette_mode:
        png = quantize_to_8bit(cropped, 256 colors)
    else:
        png = save_rgba_png(cropped)
    
    // 4. 写入 BDN XML 事件
    event = BDN_Event{
        in_tc:  frame_to_timecode(frame_idx),
        out_tc: frame_to_timecode(end_frame),
        graphic: { width, height, x, y, filename }
    }
    xml_writer.write_event(event)

阶段 4：BDN XML → SUP 编码

BDSup2Sub (Java) 或 SUPer (Python) 将 BDN XML + PNG 转为 PGS 二进制流：

// SUP/PGS 编码流程
for each_event(bdn_event):
    // 1. 读取 PNG 并量化为 8-bit 索引色
    image = load_png(bdn_event.graphic)
    palette = generate_palette(image, max_colors=256)
    indexed = quantize(image, palette)
    
    // 2. RLE 编码位图数据
    rle_data = rle_encode(indexed)
    
    // 3. 构建 PGS 段序列
    segments = [
        PGS_Palette_Definition(palette),
        PGS_Window_Definition(x, y, w, h),
        PGS_Object_Definition(rle_data),
        PGS_Composition(pts, object_refs)
    ]
    
    // 4. 写入 SUP 文件
    for seg in segments:
        write_pgs_segment(sup_file, seg)

2.3 时间码处理流程

时间码精度是关键 ASS 使用毫秒级时间戳（00:00:01.234），而 PGS 使用帧级时间码（00:00:01:12 @23.976fps）。转换必须精确到帧边界，否则会导致字幕提前/延迟显示。ass2bdnxml v0.4+ 已将基础时间单位从毫秒改为帧。

3. 核心格式深度解析

3.1 ASS (Advanced SubStation Alpha) 格式

; ASS 文件结构
[Script Info]
Title: Example
ScriptType: v4.00+
PlayResX: 1920
PlayResY: 1080

[V4+ Styles]
Format: Name,Fontname,Fontsize,PrimaryColour,...
Style: Default,Arial,48,&H00FFFFFF,...

[Events]
Format: Layer,Start,End,Style,Name,MarginL,MarginR,MarginV,Effect,Text
Dialogue: 0,0:00:01.00,0:00:03.00,Default,,0,0,0,,Hello World

ASS 渲染复杂度来源：

  • 样式系统：字体、颜色（&HAABBGGRR）、边框、阴影、透明度
  • 位置控制：\pos(x,y)、\move(x1,y1,x2,y2)、\an 对齐
  • 动画标签：\t() 渐变、\fad() 淡入淡出、\frz 旋转
  • 高级特效：卡拉OK \k/\kf、遮罩 \clip、绘图 \p
  • 字体覆盖：\fn 切换字体、\fs 改变大小

3.2 BDN XML 格式

BDN (Blu-ray Disc Network) XML 是 Sony 定义的中间格式，描述字幕事件的元数据：

<?xml version="1.0" encoding="UTF-8"?>
<BDN Version="0.93">

    <Events LastEventOutTC="01:30:00:00" NumberofEvents="1500" Type="Graphic"/>
  </Description>
  <Events>
    <Event Forced="False" InTC="00:00:01:00" OutTC="00:00:03:00">
      <Graphic Width="200" Height="50" X="860" Y="900">
        00000000.png
      </Graphic>
    </Event>
  </Events>
</BDN>

3.3 SUP/PGS (Presentation Graphic Stream) 格式

PGS 是蓝光规范定义的位图字幕格式，由一系列 Display Set 组成：

PGS 段类型

类型码 名称 功能
0x14 PGS_PALETTE (PDS) 定义 256 色调色板（YCbCrA 格式）
0x15 PGS_OBJECT (ODS) 定义位图对象（RLE 编码）
0x16 PGS_PG_COMPOSITION (PCS) 定义页面组合（对象位置、窗口引用）
0x17 PGS_WINDOW (WDS) 定义显示窗口区域
0x80 PGS_END_OF_DISPLAY (END) 标记 Display Set 结束

Display Set 结构

// 一个完整的 Display Set 由以下段组成：
Display_Set {
    PCS  (Page_Composition_Segment)   // 页面组合
    WDS* (Window_Definition_Segment)  // 窗口定义（可多个）
    PDS* (Palette_Definition_Segment) // 调色板定义
    ODS* (Object_Definition_Segment)  // 位图对象（可多个）
    END  (End_of_Display_Segment)     // 结束标记
}

// PGS 段头结构 (13 bytes)
struct PGS_Segment_Header {
    magic:       [u8; 2],  // 0x50 0x47 ("PG")
    pts:         u32,       // Presentation Timestamp (90kHz)
    dts:         u32,       // Decode Timestamp
    segment_type: u8,      // 段类型
    segment_size: u16,     // 段数据大小
}

Object Definition Segment (ODS) — 位图数据

// ODS 数据结构
struct ODS_Data {
    object_id:    u16,       // 对象 ID
    object_version: u8,     // 版本号
    last_in_sequence: u8,   // 是否为最后一个分片
    data_length:  u32,       // 数据长度
    width:        u16,       // 图像宽度
    height:       u16,       // 图像高度
    rle_data:     [u8],      // RLE 编码的位图数据
}

// RLE 编码规则：
// - 0x00 + count + pixel: 重复 count+1 次
// - 0x40 + count + p1 + p2: 重复 count+1 次的 2-color 模式
// - pixel: 单个像素
关键约束：PGS 解码器缓冲区 蓝光规范对 PGS 解码器有严格的缓冲区限制（通常 2MB）。大型字幕图像（如带特效的全屏字幕）必须进行图像分割或优化，否则会导致播放卡顿或解码失败。easyavs2bdnxml 的"双窗口"特性正是为了解决这个问题。

4. 关键模块代码思维分析

4.1 ASS 解析与渲染 (libass)

libass 是整个管线的核心渲染引擎。调用流程：

// libass API 调用序列
// 1. 初始化
ASS_Library* lib = ass_library_init();
ASS_Renderer* renderer = ass_renderer_init(lib);
ass_set_frame_size(renderer, 1920, 1080);
ass_set_fonts(renderer, NULL, "sans-serif", 1, NULL, 1);

// 2. 加载 ASS 文件
ASS_Track* track = ass_read_file(lib, "subtitle.ass", NULL);

// 3. 逐帧渲染
for (ts = 0; ts < duration; ts += frame_dur) {
    int changed;
    ASS_Image* img = ass_render_frame(renderer, track, ts, &changed);
    
    // img 是一个链表，每个节点是一个 RGBA 位图
    while (img) {
        // img->w, img->h: 图像尺寸
        // img->stride: 行字节数
        // img->bitmap: 8-bit alpha 值（预乘 alpha）
        // img->color: RGB 颜色 (0xRRGGBB)
        // img->dst_x, img->dst_y: 在屏幕上的位置
        blend_bitmap(canvas, img);
        img = img->next;
    }
}

ASS_Image 混合算法

// 将 ASS_Image 混合到 RGBA 画布
void blend_bitmap(uint8_t* canvas, ASS_Image* img) {
    uint8_t r = (img->color >> 24) & 0xFF;
    uint8_t g = (img->color >> 16) & 0xFF;
    uint8_t b = (img->color >> 8)  & 0xFF;
    
    for (y = 0; y < img->h; y++) {
        for (x = 0; x < img->w; x++) {
            uint8_t alpha = img->bitmap[y * img->stride + x];
            int dx = img->dst_x + x;
            int dy = img->dst_y + y;
            
            // Alpha 混合（预乘 alpha）
            int idx = (dy * canvas_w + dx) * 4;
            canvas[idx+0] = lerp(canvas[idx+0], r, alpha); // R
            canvas[idx+1] = lerp(canvas[idx+1], g, alpha); // G
            canvas[idx+2] = lerp(canvas[idx+2], b, alpha); // B
            canvas[idx+3] = min(255, canvas[idx+3] + alpha); // A
        }
    }
}

4.2 帧差分与事件检测

avs2bdnxml 的核心优化——只输出有变化的帧：

// 帧差分算法
prev_frame = NULL;
event_start = -1;

for each frame:
    current = render_frame(frame_idx);
    
    if frames_equal(prev_frame, current):
        // 帧相同，延长当前事件的 out_tc
        extend_current_event(frame_idx);
    else:
        // 帧不同，结束旧事件，开始新事件
        if event_start >= 0:
            finalize_event(event_start, frame_idx);
        event_start = frame_idx;
        save_png(current);
    
    prev_frame = current;

4.3 颜色量化

将 32-bit RGBA 图像量化为 8-bit 索引色（256 色调色板）：

// 量化流程
1. 统计图像中使用的颜色
2. 如果颜色数 <= 256:
    - 直接创建调色板
3. 如果颜色数 > 256:
    - 使用中位切分法 (Median Cut) 或八叉树量化
    - 将颜色聚类为 256 个代表性颜色
    - 为每个像素分配最近的调色板索引
4. 生成 8-bit 索引 PNG

// 常用量化算法：
- Median Cut: 递归将颜色空间切分为 256 个区域
- Octree: 八叉树结构聚合相近颜色
- K-Means: 聚类算法（计算量大但质量好）
- libimagequant: 高质量量化库（cubicibo/ass2bdnxml 使用）

4.4 PGS RLE 编码

// PGS RLE 编码实现
Vec<u8> rle_encode(&[u8] indexed_pixels) {
    let mut result = Vec::new();
    let mut i = 0;
    
    while i < pixels.len() {
        // 计算连续相同像素的长度
        let pixel = pixels[i];
        let mut count = 1;
        while i + count < pixels.len() 
              && pixels[i + count] == pixel 
              && count < 0xFF {
            count += 1;
        }
        
        if count > 2 {
            // 长重复：0x00 + count + pixel
            result.push(0x00);
            result.push(count as u8);
            result.push(pixel);
        } else {
            // 短重复或单像素：直接输出
            for _ in 0..count {
                result.push(pixel);
            }
        }
        i += count;
    }
    result
}

5. easyavs2bdnxml 的设计哲学

5.1 核心设计思想

分离关注点 (Separation of Concerns)

easyavs2bdnxml 采用"胶水层"架构，每个组件只负责一件事：

  • AviSynth: 脚本引擎，负责帧的获取和处理
  • VSFilter: 专业字幕渲染器，处理所有 ASS 特效
  • avs2bdnxml: 格式转换器，负责帧→XML+PNG
  • BDSup2Sub: SUP 编码器，处理 PGS 规范细节
  • easyavs2bdnxml: 编排器，负责流程自动化
设计优势 这种"组合式"设计允许各组件独立升级。例如，可以用 VSFilterMod 替换 VSFilter 以获得更好的特效支持，或用 SUPer 替换 BDSup2Sub 以获得更好的压缩率。

渐进式降级 (Progressive Degradation)

管线设计考虑了不同场景的兼容性：

  • 完整模式：AviSynth + VSFilter → 最高保真度
  • 简化模式：libass 直接渲染 → 跨平台但可能有差异
  • 双窗口：同一时间戳存储两张图 → 原盘 DIY 兼容
  • 单窗口：ASS2SUP 模式 → 电脑播放兼容性更好

5.2 关键技术决策

决策 原因 权衡
使用 AviSynth 而非直接解析 AviSynth 生态成熟，VSFilter 只支持 DirectShow Windows 依赖，安装复杂
中间格式 BDN XML+PNG 标准化格式，可被多种工具消费 文件数量多，占用磁盘空间
支持 VSFilterMod VSFilterMod 支持更多 ASS 特效 兼容性不如原版 VSFilter
双窗口存储 原盘规范要求，防止播放卡顿 SUP 文件体积增大
Java 的 BDSup2Sub 跨平台，功能完善 需要 JRE，性能不如原生代码

5.3 "大金过水" 机制

什么是"大金过水"？ 在原盘 DIY 社区，"大金过水"指将字幕图像分割为多个较小的窗口（Window），以避免单个 PGS 对象过大导致蓝光播放器缓冲区溢出。easyavs2bdnxml 的双窗口特性正是为此设计——将一个字幕事件拆分为两个独立的图形对象，分别放置在不同的窗口区域。

6.3 两种实现路径

路径 A：FFI 绑定 libass（推荐）

优势 复用 libass 成熟的渲染引擎，避免重新实现复杂的 ASS 特效系统。libass 是经过 15+ 年打磨的工业级代码，处理了大量边界情况和兼容性问题。 
// 路径 A: FFI 绑定 libass
extern "C" {
    fn ass_library_init() -> *mut ASS_Library;
    fn ass_renderer_init(lib: *mut ASS_Library) -> *mut ASS_Renderer;
    fn ass_render_frame(
        renderer: *mut ASS_Renderer,
        track: *mut ASS_Track,
        now: i64,
        detect_change: *mut i32
    ) -> *mut ASS_Image;
}

// Rust 安全封装
pub struct AssRenderer {
    lib: *mut ASS_Library,
    renderer: *mut ASS_Renderer,
}

impl AssRenderer {
    pub fn render_frame(&self, track: &AssTrack, ts_ms: i64) -> Vec<AssImage> {
        let mut changed = 0;
        let raw = unsafe { 
            ass_render_frame(self.renderer, track.ptr, ts_ms, &mut changed) 
        };
        // 将 ASS_Image 链表转为 Rust Vec
        collect_ass_images(raw)
    }
}

路径 B：纯 Rust 实现渲染器

挑战 需要完整实现 ASS 的文本排版、字体渲染、特效系统。这是最大的工程量，但也提供了最大的控制力和可移植性。

6.4 可行性结论

结论：完全可行，推荐路径 A
  • 路径 A (FFI 绑定 libass) 可在 2-3 个月内完成 MVP
  • 路径 B (纯 Rust) 需要 6-12 个月，但长期价值更高
  • 可采用渐进策略：先用路径 A 快速交付，再逐步替换为纯 Rust 组件

7. Rust 实现难点与解决方案

7.1 难点一：ASS 文本渲染

问题描述

ASS 格式支持极其丰富的文本特效，包括但不限于：

  • 多字体、多大小、多颜色混排
  • \t() 渐变动画（任意属性的平滑过渡）
  • \clip() / \iclip() 矩形和贝塞尔曲线遮罩
  • \p 绘图模式（ASS 绘图命令）
  • \k / \kf / \ko 卡拉OK 效果
  • \be / \blur 高斯模糊
  • \shad / \xshad / \yshad 阴影
  • \frx / \fry / \frz 3D 旋转

解决方案

方案 实现 优劣
FFI 调用 libass libass-sys crate 最可靠，但需要编译 C 库
Skia 渲染 skia-safe crate 功能强大，但二进制体积大
逐步实现 fontdue + 自研特效 完全可控，但工作量巨大

7.2 难点二：字体处理

问题描述

  • ASS 可以指定任意字体名，需要系统字体枚举
  • ASS 文件可以内嵌字体（fontname.ttf base64）
  • 需要处理字体回退（fallback）机制
  • 需要支持 OpenType 特性（连字、字距调整等）
